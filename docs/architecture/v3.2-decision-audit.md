# Neuron v3.2 Decision Audit

> Systematic crate-by-crate audit of Neuron implementations against
> Agentic Decision Framework v3.2. Generated 2026-03-03.

## Summary

### Protocol Layer (layer0)
**Decisions:** D1-D5, C1-C5, L1-L5
Audit of neuron/layer0 crate against Agentic Decision Framework v3.2. The crate implements core protocol traits well but has six actionable gaps: (1) HookPoint enum missing variants for D3C cognitive retry, C1/C2 delegation boundaries, and L1 memory write observation; (2) ExitReason lacks structural priority ordering per v3.2 layered exit spec; (3) StateStore protocol doesn't expose D2C hot/warm/cold tier metadata; (4) BudgetEvent doesn't emit step/loop/wall-clock granularity required by L4; (5) CompactionEvent missing outcome/result variant for observing compaction success; (6) Effect enum could benefit from explicit delegation-result type. Severity ranges from medium (missing hook points) to low (nice-to-have observability).

### Turn Engine (op-react, turn, turn-kit, tool, context)
**Decisions:** D2B, D2D, D2E, D3A, D3C, D4C, D5, L4
Audit of Neuron crates (neuron-op-react, neuron-turn, neuron-turn-kit, neuron-tool, neuron-context) against v3.2 Agentic Decision Framework reveals 7 sub-optimal findings across 8 decision points. Key gaps: (1) D5 exit priority is inverted (checks max_turns before safety hook), (2) D2E missing compaction reserve guarantee, (3) D3A lacks task-type routing, (4) D3C has no cognitive retry distinction, (5) D4C tool backfill lacks formatting options, (6) L4 budget spectrum incomplete (no step/loop limits), (7) Hook composition ignores v3.2 type-based dispatch (all short-circuit).

### Providers (anthropic, openai, ollama)
**Decisions:** D3A, D3C, D2E
Audit of three provider crates (anthropic, openai, ollama) against Agentic Decision Framework v3.2. Key findings: (1) D3A—no task-type routing; all providers hardcode single default model with no complexity-based selection. (2) D3C—error types conflate retryable and non-retryable failures; no explicit safety-refusal or budget-exhaustion non-retry enforcement. (3) D2E—token usage reporting works, but no pre-flight token counting interface or budget enforcement. Anthropic/OpenAI map ContentFilter correctly but Ollama does not. All providers disable streaming (stream=false hardcoded).

### Orchestration + Effects (orch-kit, orch-local, effects-core, effects-local)
**Decisions:** D3B, L1, L2, L3, C1-C4
Audit of four Neuron crates (orch-kit, orch-local, effects-core, effects-local) against Agentic Decision Framework v3.2. Findings: 6 sub-optimal patterns identified across D3B (durability), L1 (memory write hooks), L2 (compaction), C2 (result return), and C3 (lifecycle). No issues with C1 (task-only injection enforced by protocol design). C4 in-memory limitation is expected for local implementations. Root causes: (1) no durability abstraction despite design promising pluggability, (2) no hook integration before memory writes, (3) no compaction/summarization strategy, (4) long-lived agents by default (v3.2 default is ephemeral), (5) no token-efficient summary return path. Most findings are MEDIUM severity (implementable without breaking changes). D3B/L3 durability is HIGH but appropriate for local-only scope.

### State + Environment + Credentials (state-*, env-local, secret-*, auth, crypto)
**Decisions:** D2C, D4A, D4B, L1
Audit of 7 Neuron crates against Agentic Decision Framework v3.2. Found 3 high-severity gaps where implementations deviate from v3.2 decisions: (1) D2C memory tiering not implemented—current state stores use flat key-value with no hot/warm/cold distinctions; (2) D4A isolation spectrum unsupported—LocalEnv hardcoded to in-process only, no mechanism to select from isolation spectrum; (3) L1 pre-compaction flush unavailable—StateStore interface lacks lifecycle awareness, cannot respond to PreCompactionFlush events. One medium-severity gap: (4) D4B Vault-mediated credentials stubbed but not implemented. Positive findings: secret leakage properly handled with sanitized error messages and scoped exposure patterns; multi-agent namespacing works correctly with Scope-based isolation.

### MCP + Hooks + Security (mcp, hooks, hook-security)
**Decisions:** C4, D4B, D2D, hooks
Audit of neuron-mcp, neuron-hooks, and neuron-hook-security crates against v3.2 Agentic Decision Framework. Key findings: (1) C4 MCP implementation covers tool operations but missing resource/prompt protocol extensions; (2) Hook composition architecture universally applies short-circuit pattern instead of type-specific patterns (guardrail→short-circuit, transformer→chain, observer→parallel); (3) Security hooks lack comprehensive coverage of dynamic credentials and all tool types; (4) Hook error observability incomplete — errors are silently swallowed without proper logging/tracing; (5) Hook registry lacks runtime mutation (remove/replace) and priority mechanisms; (6) Security hooks cannot register at PreInference hook point for prompt injection detection.

## Detailed Findings

### Protocol Layer (layer0)

**`layer0/src/hook.rs` L11-L22**
HookPoint enum missing 4+ variants needed for v3.2 hook coverage. Current: 6 points (PreInference, PostInference, PreToolUse, PostToolUse, ExitCheck, ToolExecutionUpdate). Missing: D3C (cognitive retry), C1 (delegation result), C2 (delegation invocation), L1 (memory write observation).

**`layer0/src/operator.rs` L96-L110**
ExitReason enum lacks priority ordering per v3.2 layered exit spec. v3.2 mandates priority: safety halt > budget > max turns > model done. Current enum is unordered. Variants present but no structural hierarchy visible.

**`layer0/src/state.rs` L3-L50**
StateStore protocol doesn't expose D2C memory tier metadata. Scope enum (Session/Workflow/Agent/Global) exists but doesn't distinguish hot/warm/cold/structural tiers. No way for implementations to tag reads/writes as tier-specific.

**`layer0/src/lifecycle.rs` L29-L60**
BudgetEvent enum lacks L4 step/loop/wall-clock granularity. Only emits cost (CostIncurred, BudgetWarning, BudgetAction). Missing events for: max_steps_reached, max_loops_reached, wall_clock_timeout. These are observable in ExitReason but not in lifecycle events.

**`layer0/src/lifecycle.rs` L69-L103**
CompactionEvent missing outcome/result variant. Current variants: ContextPressure, PreCompactionFlush, CompactionComplete, ProviderManaged. Missing: observation of compaction outcome (did it work? how much space saved? any errors?). Only has success path, no failure/retry path.

**`layer0/src/effect.rs` L23-L65**
Effect enum could benefit from explicit delegation-result type for C1 observability. Current: WriteMemory, DeleteMemory, Signal, Delegate, Handoff, Log, Custom. Handoff is terminal. Missing: DelegationResult or UpdateDelegationState for non-terminal delegation responses.

**Risks:**
- HookPoint enum is not extensible enough for D3C cognitive retry policies. Current design assumes hooks fire at model/tool boundaries but doesn't support hooking the retry decision loop itself. This prevents policy-driven retry strategies (exponential backoff, circuit breaker, jitter) from being observable.
- ExitReason doesn't structurally encode the v3.2 layered exit priority. An implementation could violate the priority by checking max_turns before budget, or timeout before observer halt. Without structural encoding, there's no way to audit compliance at the type level.
- StateStore protocol lacks hot/warm/cold tier information. Implementations can't expose which tier a read was serviced from or allow callers to request specific tiers. This breaks D2C contract — the memory tier decision is made but not observable to the operator runtime.
- Missing BudgetEvent variants for step/loop/wall-clock granularity means orchestrators can't easily observe when step limits or loop limits are approaching — they only get cost events. This creates blind spots for proactive budget management.
- CompactionEvent only models the success path. If compaction fails or is skipped, there's no event to indicate why. Multi-agent systems need to coordinate on compaction outcomes (did your state get flushed? can we proceed?). Current design can't represent negative outcomes.
- Delegation observability gap: An operator delegating to a sub-agent gets back a result, but there's no Effect type that directly represents 'I want to persist this delegation result' as a distinct concern from Signal (peer comms) or WriteMemory (arbitrary state). C1/C2 require explicit delegation boundaries but Effect enum treats all agent communication as either Delegate/Handoff (control) or Signal (asynchronous). Missing: intermediate result persistence for delegation chains.

### Turn Engine (op-react, turn, turn-kit, tool, context)

**`op/neuron-op-react/src/lib.rs` L988-L1052**
Exit priority ordering: checks max_turns, then budget, then timeout, then hook. Should be: safety (hook) > budget > turns > done

**`op/neuron-op-react/src/lib.rs` L1065-L1069**
Context compaction: no reserve guarantee. Line 1067 sets limit = max_tokens * 4 but never ensures non-zero reserve buffer remains after compaction

**`op/neuron-op-react/src/lib.rs` L183-L200**
Model selection: hardcoded to single model (config.model). No task-type routing based on complexity, input characteristics, or OperatorInput.metadata

**`op/neuron-op-react/src/lib.rs` L410-L420**
Retry handling: only infrastructure retry (is_retryable check). No cognitive retry pattern (re-reason with failure context) distinct from simple replay

**`op/neuron-op-react/src/lib.rs` L567-L575**
Tool backfill: tool results formatted as plain ContentPart::ToolResult with raw content string. No options for compacted format or code-mediated backfill as v3.2 specifies

**`op/neuron-op-react/src/lib.rs` L60-L67**
Budget tracking: only max_turns, max_cost, max_duration. Missing step limits, loop limits, and per-tool execution budgets that v3.2 adds to spectrum

**`hooks/neuron-hooks/src/lib.rs` L41-L57**
Hook dispatch: all hooks short-circuit on non-Continue actions. v3.2 requires type-based composition: guardrail=short-circuit, transformer=chain (apply all), observer=parallel. Current implementation lacks hook type awareness

**`turn/neuron-turn/src/context.rs` L1-L50**
ContextStrategy trait: no reserve field or parameter. No mechanism to enforce minimum buffer (e.g., 10-15% of limit) before compaction triggers. v3.2 says reserve must never be zero

**Risks:**
- D5 inversion: Exit checks max_turns before safety hook, violating v3.2 priority. Safety halt should fire first, preventing runaway loops if hook detects anomaly.
- D2E gap: No compaction reserve. Context can be compacted to exactly max_tokens, leaving zero buffer for model response. Provider truncation may cascade.
- D3A missing: Single model per operator. Complex tasks (reasoning vs. lookup vs. planning) should route to different models; current design forces one model for all task types.
- D3C gap: Cognitive retry unimplemented. Failed tool calls trigger retryable infrastructure error, not re-reasoning with failure narrative. Misses opportunity to learn from failures.
- D4C underspecified: Tool backfill is raw string only. No compacted JSON format option, no code-mediated transformation hooks (e.g., redaction, summarization).
- L4 incomplete: Budget spectrum lacks step/loop granularity. Only max_turns exists; no per-tool execution budgets, no step-wise halting, no loop iteration limits within a turn.
- Hook composition static: All hooks short-circuit. No distinction between guardrail (safety-critical, short-circuit) vs. transformer (apply all modifications) vs. observer (parallel, non-blocking). Orchestrators cannot compose hooks by type.
- ContextStrategy contract loose: No contractual guarantee of reserve buffer. SlidingWindow and NoCompaction implementations maintain no minimum reserve, risking context overflow.

### Providers (anthropic, openai, ollama)

**`provider/neuron-provider-anthropic/src/lib.rs` L34-L40**
Model selection: hardcoded default to claude-haiku-4-5-20251001 when none provided; no task-type routing

**`provider/neuron-provider-anthropic/src/lib.rs` L138-L163**
Error handling: maps HTTP errors to ProviderError variants but does not check for safety refusals; no distinction between retryable/non-retryable in response body

**`provider/neuron-provider-anthropic/src/lib.rs` L78-L84**
StopReason parsing: does not map to ContentFilter; only EndTurn, ToolUse, MaxTokens handled

**`provider/neuron-provider-anthropic/src/lib.rs` L92-L102**
Token usage: reports input_tokens, output_tokens, cache tokens; includes cost calculation but no budget pre-check

**`provider/neuron-provider-openai/src/lib.rs` L50-L56**
Model selection: hardcoded default to gpt-4o-mini; no task-type routing or complexity-aware routing

**`provider/neuron-provider-openai/src/lib.rs` L253-L256**
StopReason parsing: correctly maps content_filter finish_reason to ContentFilter, but ContentFilter is not enforced as non-retryable in retry logic

**`provider/neuron-provider-openai/src/lib.rs` L230-L266**
Error handling: similar to Anthropic; maps HTTP status to ProviderError but RequestFailed is over-broad (includes both transient and non-retryable errors)

**`provider/neuron-provider-openai/src/lib.rs` L242-L250**
Token usage: reports via prompt_tokens, completion_tokens, cached_tokens; no pre-flight budget interface or validation against D2E allocation percentages

**`provider/neuron-provider-ollama/src/lib.rs` L60-L68**
Model selection: hardcoded default to llama3.2:1b; no task-type routing

**`provider/neuron-provider-ollama/src/lib.rs` L174-L175**
Streaming disabled: stream field hardcoded to false; no streaming support

**`provider/neuron-provider-ollama/src/lib.rs` L214-L230**
StopReason parsing: does not handle ContentFilter at all (Ollama may not report it); only maps stop/length to EndTurn/MaxTokens

**`provider/neuron-provider-ollama/src/lib.rs` L189-L210**
Error handling: identical pattern to Anthropic/OpenAI; no safety-refusal detection

**`provider/neuron-provider-ollama/src/lib.rs` L233-L241**
Token usage: uses prompt_eval_count/eval_count; no pre-flight budget checks or allocation validation

**`turn/neuron-turn/src/provider.rs` L37-L45**
ProviderError.is_retryable(): marks RateLimited and RequestFailed as retryable, but RequestFailed is too broad—includes timeouts (should retry) and network errors, but v3.2 says safety refusals and budget exhaustion should NOT retry

**`turn/neuron-turn/src/types.rs` L116-L126**
StopReason enum: includes ContentFilter variant, but operators must manually enforce it as non-retryable; no built-in constraint at provider level

**Risks:**
- D3A violation: No task-type routing. Provider selection is per-request but model selection within each provider is hardcoded default. For agentic systems routing by task complexity (explore=Haiku, plan=Sonnet, edit=Opus), this requires application-level wrapping rather than built-in provider support. Cost efficiency loss: system overpays for simple tasks (e.g., using Opus for fact retrieval).
- D3C violation: RequestFailed conflates retryable (timeout, transient network) and non-retryable (budget exhausted, safety refusal). When ContentFilter is detected (OpenAI provider does this), it produces a ProviderResponse with ContentFilter stop_reason, but the caller must manually enforce non-retry—the provider error classification does not capture this constraint. v3.2 explicitly mandates: Budget-exceeded errors should NOT retry. Safety refusals should NOT retry.
- D3C violation: Anthropic provider does not map to ContentFilter stop_reason (no Anthropic API endpoint returns content_filter as stop_reason string). If Anthropic implements content filtering via a different mechanism (response body field, metadata), this would be silently missed. No code comment documenting this gap.
- D3C violation: Ollama provider does not map to ContentFilter at all. Ollama's API may not support structured safety refusal reporting, but this is not documented or handled specially. If Ollama adds this capability, it would be silently unobserved.
- D2E violation: No pre-flight token counting interface. Providers report consumed tokens post-completion, but the turn engine has no way to predict token usage before sending a request. This prevents budget-aware context assembly (e.g., can I fit the next tool output in remaining budget?). v3.2 specifies ~10% system, ~67.5% conversation+results budget, but there is no validation or enforcement of these percentages.
- Design: stream field is hardcoded to false in all providers. v3.2 notes that streaming observation should be read-only (ToolExecutionUpdate pattern). Current design precludes streaming support entirely—if streaming is added later, observer hooks would need to be retrofitted into Provider trait.
- Error classification: AuthFailed is correctly non-retryable, but the distinction between transient network errors (retry) and model overload (retry with backoff) is not made. A 503 Service Unavailable maps to RequestFailed, which is retryable, but without backoff strategy or provider-level backoff support, aggressive retries could worsen cascading failures.
- Anthropic/OpenAI: response.cost is calculated post-hoc with hardcoded per-token rates. These rates are inline comments only (not versioned configuration). If pricing changes (as it frequently does), code must be updated and re-deployed. v3.2 does not address pricing strategy, but this is a maintenance risk for production systems.

### Orchestration + Effects (orch-kit, orch-local, effects-core, effects-local)

**`orch/neuron-orch-kit/src/runner.rs` L1-L50**
OrchestratedRunner and LocalEffectInterpreter: Effect interpretation with no durability, no PreMemoryWrite hook, no compaction strategy, no summary return path

**`orch/neuron-orch-kit/src/runner.rs` L80-L130**
LocalEffectInterpreter.execute_effect(): Direct state.write() call without PreMemoryWrite hook

**`orch/neuron-orch-kit/src/runner.rs` L190-L250**
OrchestratedRunner.run(): max_followups safety bound but no compaction/summarization before exceeding

**`orch/neuron-orch-local/src/lib.rs` L1-L50**
LocalOrch documentation: Explicitly states 'No durability — operators that fail are not retried' (D3B/L3 issue)

**`orch/neuron-orch-local/src/lib.rs` L40-L50**
LocalOrch.register(): Agents stored in HashMap with no lifecycle management (C3 issue: long-lived by default)

**`effects/neuron-effects-local/src/lib.rs` L60-L110**
LocalEffectExecutor.execute(): Direct state.write() call without PreMemoryWrite hook (L1 issue)

**Risks:**
- D3B/L3 (Durability/Crash Recovery): Zero durability is appropriate for local development but insufficient for production multi-agent long-running tasks. v3.2 notes that durability is essential for 500+ iteration workflows. No abstraction exists to swap in checkpoint/event-replay implementations without rewriting effect executors. Risk: crashes at iteration N>>100 cause total context loss.
- L1 (PreMemoryWrite hook): No hook fires before state.write(). v3.2 identifies PreMemoryWrite as medium-priority for redaction/validation. Risk: sensitive data written without interception; no policy enforcement on what persists.
- L2 (Compaction): max_followups safety bound stops execution at 128 followups but doesn't summarize old turns. v3.2 pattern: 'write state before compaction, then compact, then resume with summary'. Current code terminates with KitError::Safety. Risk: long-running composite tasks fail instead of compacting intelligently.
- C2 (Result return): Delegate result is the full OperatorOutput in ExecutionTrace. No summary path for token efficiency. v3.2 shows Claude Code uses summary-only return to parent. Risk: parent context bloat when delegating to expensive sub-agents.
- C3 (Lifecycle): Agents registered via HashMap persist indefinitely. No ephemeral/fire-and-forget lifecycle. v3.2 confirms ephemeral is default (Claude Code sub-agents, CrewAI tasks). Risk: agents accumulate in registry, resource leaks in long-lived orchestrators, no task-scoped cleanup.
- Hook architecture: Neither orch-kit nor effects-local implement any hooks (PreMemoryWrite, PreComposition, PostComposition missing). v3.2 shows 9/23 decisions are fully hookable. Current architecture blocks future addition of guardrails (e.g., exfil prevention), transformers (redaction), observers (audit logging) without rewriting effect executors.

### State + Environment + Credentials (state-*, env-local, secret-*, auth, crypto)

**`state/neuron-state-memory/src/lib.rs` L1-L120**
In-memory state store implementation - uses flat key-value HashMap with no tiering (hot/warm/cold). Implements StateStore trait with read/write/delete/list/search, but no memory tier distinction.

**`state/neuron-state-fs/src/lib.rs` L1-L80**
Filesystem state store implementation - maps scopes to subdirectories, stores keys as .json files. Also flat key-value with no tiering, no lifecycle hooks for compaction.

**`env/neuron-env-local/src/lib.rs` L1-L50**
Local environment implementation - explicitly 'local mode only' with in-process execution. No support for isolation boundary selection from EnvironmentSpec (Process/Container/Gvisor/MicroVM/Wasm/NetworkPolicy/Custom).

**`secret/neuron-secret/src/lib.rs` L1-L80**
Base secret resolver trait - defines SecretResolver, SecretValue (with scoped exposure), SecretRegistry. Registry dispatches by source type but no Vault integration.

**`secret/neuron-secret-vault/src/lib.rs` L1-L40**
Vault resolver stub - VaultResolver trait impl always returns BackendError('is a stub'). Vault-mediated dynamic credentials not implemented.

**`env/neuron-env-local/src/lib.rs` L135-L180**
Credential injection logic - sanitize_secret_error() converts errors to safe strings (no secret leakage). EnvVar and File injection implemented; Sidecar is no-op.

**`layer0/src/lifecycle.rs` L80-L110**
PreCompactionFlush lifecycle event defined - expects agent to flush critical state before compaction. No mechanism in state stores to receive or respond to this event.

**`layer0/src/environment.rs` L40-L80**
IsolationBoundary enum defined with 7 isolation types (Process/Container/Gvisor/MicroVM/Wasm/NetworkPolicy/Custom). LocalEnv does not consume this enum.

**`layer0/src/state.rs` L1-L60**
StateStore trait interface - deliberately minimal (no lifecycle hooks). Compaction coordination happens via separate Lifecycle interface, not through StateStore methods.

**Risks:**
- D2C (Memory Tiering) — Production systems will not be able to implement hot/warm/cold memory strategies defined in v3.2. State stores treat all data equally, causing unnecessary context pollution. The StateStore trait cannot distinguish access patterns or cache strategies. High risk for long-running agents where context compaction is critical.
- D4A (Isolation Spectrum) — Only local in-process execution supported. Production deployments requiring container, gVisor, or MicroVM isolation cannot be built. EnvironmentSpec.isolation field defined in layer0 but completely ignored by LocalEnv. No path to add container or remote execution implementations without replacing entire environment layer. High risk for multi-tenant or untrusted-code execution scenarios.
- L1 (Pre-Compaction Flush) — StateStore interface has no lifecycle awareness. When layer4 orchestrator fires PreCompactionFlush event, no mechanism exists for state stores to respond. Critical state cannot be flushed before old turns are destroyed. Loses important agent memories during compaction. High risk for agents relying on durable memory across context boundaries.
- D4B (Vault Implementation Gap) — VaultResolver is a stub. Vault-mediated dynamic credentials (v3.2's recommended approach) are not implemented. Only static credentials via environment variables or files are currently supported. Medium risk for security-conscious deployments needing ephemeral, task-scoped credentials.
- State Store Passivity — Both MemoryStore and FsStore are passive backends. They cannot participate in lifecycle coordination or implement preservation strategies. This violates the principle that lifecycle concerns span all protocols. May require redesigning StateStore interface to accept lifecycle context or callbacks.
- Missing Hook Integration Points — Pre-compaction flush is a lifecycle event, not a state store method. For state stores to implement pre-flush, they need either (1) to receive PreCompactionFlush events via callback, (2) to have lifecycle-aware interface extensions, or (3) to be wrapped by a lifecycle-aware proxy. Current design creates a coordination gap.

### MCP + Hooks + Security (mcp, hooks, hook-security)

**`turn/neuron-mcp/src/lib.rs` L1-L17**
MCP client/server bridge. Implements tool discovery and invocation but missing resource and prompt protocol support per C4 v3.2.

**`turn/neuron-mcp/src/client.rs` L70-L110**
McpClient::discover_tools() - tool discovery implementation. No resource or prompt discovery methods.

**`hooks/neuron-hooks/src/lib.rs` L35-L60**
HookRegistry::dispatch() - applies universal short-circuit pattern to all hook actions, regardless of type. Should vary by hook class per v3.2 constitution.

**`hooks/neuron-hooks/src/lib.rs` L12-L25**
HookRegistry structure and behavior. No runtime mutation methods (remove/replace), no ordering/priority mechanism.

**`hooks/neuron-hook-security/src/lib.rs` L1-L50**
RedactionHook and ExfilGuardHook - only two hook implementations. Pattern-based redaction for AWS/Vault/GitHub tokens, no vault-mediated dynamic credential support.

**`hooks/neuron-hook-security/src/lib.rs` L70-L120**
ExfilGuardHook detection logic limited to shell commands (curl/wget). Doesn't generically cover all tool types per D2D.

**`hooks/neuron-hooks/src/lib.rs` L47-L55**
Hook error handling - errors logged (commented) but not actually traced/logged to any system.

**Risks:**
- Hook composition bug: all actions short-circuit, violating v3.2 typed composition (guardrails should short-circuit, transformers should chain, observers should parallel). This means ModifyToolInput hooks would incorrectly halt pipeline if followed by other hooks.
- Security hook incompleteness: RedactionHook only redacts static patterns, not vault-mediated dynamic credentials per D4B. No support for credential lifecycle (e.g., token expiration).
- Exfiltration detection too narrow: ExfilGuardHook only checks shell commands (curl/wget), missing detection for other tool types (MCP tools, HTTP tools). Per D2D, tool definitions are source of truth, but guard doesn't generically inspect tool types.
- Missing hook points for security: PreInference hook point exists but no security hooks fire there. Prompt injection detection would require PreInference hook; currently impossible.
- Error observability gap: Hook errors are silently swallowed; comment says 'would go to tracing/logging' but no actual logging. Security hooks silently fail instead of alerting operators.
- No hook runtime mutation: HookRegistry has no remove/replace methods. Hooks cannot be disabled at runtime or priority-ordered. One-time setup only.
- C4 protocol incompleteness: neuron-mcp implements only tool discovery/invocation. MCP v2 includes resource and prompt protocols per C4 v3.2, but no resource_list() or prompt_list() methods implemented.
- Tool execution metadata not propagated: ExfilGuardHook checks tool input string, but doesn't use tool definition/schema metadata (e.g., concurrency_hint from ToolDyn::concurrency_hint()). Per D2D, tool defs are source of truth for execution metadata.
- Composition with other patterns: Hook registry assumes linear ordering. No support for conditional logic (AND/OR), priority tiers, or type-specific branching that v3.2 five-pattern taxonomy requires.
